# Glyph 实现方案

> *LLM 是代码美学的翻译官 — 它同时理解代码语义和艺术语言*

---

## 一、产品定义

**Glyph** 是一个 Web 应用：你拖入一段代码，它生成一幅有审美价值的抽象画。

不是代码高亮，不是可视化工具。是**代码的"艺术肖像"**——每一幅画都能解释"为什么你的代码长这样"。

### 用户流程

```
拖入代码文件 / 粘贴代码
        ↓
   代码被 tree-sitter 解析为 AST
        ↓
   AST 特征 + 源码 → Claude API（艺术翻译）
        ↓
   Claude 分析美学特征，生成 SVG 艺术品
        ↓
   浏览器展示画作 + 艺术家陈述（为什么这么画）
        ↓
   下载 SVG / PNG / 分享
```

### 三层管道

| 层 | 职责 | 技术 | 输入 → 输出 |
|----|------|------|------------|
| **解析层** | 提取代码结构特征 | tree-sitter WASM | 源码 → AST 特征向量 |
| **翻译层** | 美学分析与艺术参数生成 | Claude API | 特征向量 + 源码 → 艺术参数 JSON |
| **渲染层** | 将参数转为 SVG | 自研 SVG 引擎 | 艺术参数 → SVG → 画布展示 |

### 为什么用 Claude API 而不是纯算法？

| 能力 | 纯算法 | Claude API |
|------|--------|------------|
| 理解代码语义 | ❌ 只能看结构 | ✅ 理解意图和范式 |
| 色彩理论应用 | ❌ 需要硬编码 | ✅ 内置艺术史知识 |
| 构图决策 | ❌ 死板规则 | ✅ 有审美判断 |
| 生成"艺术家陈述" | ❌ 不可能 | ✅ 天然能力 |
| 风格多样性 | ❌ 几套模板 | ✅ 无限变化 |

## 三、核心引擎设计

### 3.1 解析层：提取"美学特征"

```typescript
interface CodeFeatures {
  // 结构特征
  functions: { name: string; depth: number; length: number }[];
  maxNestingDepth: number;
  nestingDistribution: number[];  // 各深度的节点数
  
  // 密度特征  
  avgLineLength: number;
  lineLengthVariance: number;
  commentRatio: number;          // 注释/代码比
  blankLineRatio: number;        // 空行/总行比
  
  // 语义特征
  variableNames: string[];      // 所有变量名
  functionNames: string[];      // 所有函数名
  dominantPatterns: string[];   // 检测到的模式（递归、管道、回调等）
  
  // 元特征
  language: string;
  totalLines: number;
  fileType: 'function' | 'class' | 'script' | 'config';
}
```

### 3.2 翻译层：LLM 作为艺术翻译官

这是整个系统的核心。Claude 接收代码和特征数据，输出结构化的艺术参数。

**Prompt 设计（两阶段）**：

**阶段一：美学分析**
```
你是一位精通代码美学和现代艺术理论的艺术家。分析以下代码，
描述它的"艺术肖像"应该是什么样。

分析维度：
1. 结构性格：这段代码是几何的、秩序的（像蒙德里安），
   还是流动的、有机的（像康定斯基）？
2. 密度节奏：哪里密集（高复杂度），哪里呼吸（空白）？
3. 色彩倾向：冷峻理性（蓝/灰调）还是温暖人性（红/金调）？
4. 文本意蕴：变量和函数名构成了怎样的语义景观？
```

**阶段二：艺术参数生成**
```
基于你的分析，生成艺术参数。输出严格 JSON：

{
  "palette": {
    "primary": "#hex",
    "secondary": ["#hex", ...],
    "accent": "#hex",
    "background": "#hex"
  },
  "composition": {
    "type": "grid | radial | diagonal | organic",
    "goldenRatio": true,
    "negativeSpacePercent": 30
  },
  "elements": [
    { "type": "rect | circle | line | polygon", 
      "x": %, "y": %, "w": %, "h": %, 
      "color": "#hex", "opacity": 0.8,
      "meaning": "这个形状代表..." }
  ],
  "artistStatement": "这幅画之所以如此，是因为..."
}
```

### 3.3 渲染层：SVG 生成

将 Claude 返回的艺术参数渲染为高质量 SVG。

- 画布默认 800×600，支持 A4/A3 打印比例
- 色彩空间使用 LAB 插值保证视觉和谐
- 添加细微纹理模拟画布质感
- 自动添加签名和日期

### 3.4 风格预设系统

除了让 Claude 自由创作，也提供明确的风格预设：

| 风格 | 触发条件 | 视觉特征 |
|------|----------|----------|
| **蒙德里安模式** | 函数式代码、Clean Architecture | 黑线网格、红蓝黄原色块 |
| **罗斯科模式** | 深层嵌套、复杂逻辑 | 大面积的浮动色域、模糊边界 |
| **波洛克模式** | 脚本、异步回调密集 | 多层线条泼溅、混沌中有秩序 |
| **极简禅模式** | 库/框架代码、接口定义 | 大面积留白、精确的单一线条 |
| **康定斯基模式** | OOP、多类交互 | 几何形状在空间中互动、暖色调 |

## 八、关键设计决策（已确定，无需讨论）

1. **LLM 作为翻译核心** — 不用纯算法，Claude 内置的艺术知识和代码理解是无可替代的
2. **确定性 + 创造性双模式** — 用 `temperature=0` 保证同代码同画作；用户也可调高温度获取变体
3. **SVG 输出优先** — 矢量、无损、可打印、可编辑，比 PNG/Canvas 更适合艺术品
4. **每幅画都有"艺术家陈述"** — 可解释性是关键差异化优势
5. **先支持 TypeScript/Python** — 最有审美表达力的两种语言
6. **配色使用 LAB 色彩空间** — 保证视觉和谐，不是 RGB 拼凑
